# 微信小程序扫码工具

一个功能强大的微信小程序扫码工具，支持二维码/条形码扫描和图书条码查询。

## 功能特性

### 1. 普通扫码模式
- 扫描二维码和条形码
- 手动输入条码内容
- 支持网址、ISBN、商品条码等多种类型
- 一键复制扫描结果

### 2. 图书扫码模式
- 扫描图书ISBN条码
- 通过API查询图书详细信息
- 显示书名、作者、出版社、索书号等信息
- 支持自定义图书API接口

### 3. 扫描批次管理
- **扫描批次概念**：每次扫描会话自动形成一个批次
- 历史记录按批次展示，便于管理
- 每个批次显示时间、数量和内容摘要
- 支持批量删除和复制整个批次

### 4. 历史记录功能
- 按批次查看历史扫描记录
- 显示批次时间、扫描数量和预览内容
- 点击进入详情页查看完整记录
- 支持批量操作（删除、复制）

### 5. 设置功能
- 配置图书查询API地址和Key
- 多语言支持（中文/英文）
- 数据导出功能
- 清空历史记录

## 技术架构

- **框架**: 微信小程序原生框架
- **语言**: JavaScript + WXSS + WXML
- **数据存储**: 本地Storage
- **API通信**: wx.request

## 核心功能说明

### 扫描批次
系统采用"扫描批次"概念管理历史记录：
- 每次进入首页开始扫描，自动创建新批次
- 同一次会话内的扫描记录归入同一批次
- 点击"完成扫描"按钮保存当前批次
- 历史记录按批次展示，便于查看和管理

### 数据流
```
扫码/输入 → 添加到当前批次 → 点击完成 → 保存批次 → 历史记录展示
```

## 使用说明

### 1. 普通扫码
1. 打开小程序，选择"普通扫码"模式
2. 点击"点击扫码"按钮或手动输入条码
3. 扫描结果自动添加到当前批次
4. 点击"完成扫描"保存批次
5. 在历史记录页查看和管理

### 2. 图书扫码
1. 切换到"图书扫码"模式
2. 在设置页配置图书API地址和Key
3. 扫描图书ISBN条码
4. 系统自动查询并显示图书信息
5. 保存到当前批次

### 3. API配置说明
在设置页配置以下参数：
- **API地址**: 图书查询接口的URL
- **API Key**: 接口认证密钥（如需要）

支持的API格式：
- GET请求
- 参数: `barcode`（条码）, `apikey`（密钥）
- 返回: JSON或XML格式图书数据

### 4. 历史记录管理
1. 进入"历史"标签页查看所有批次
2. 点击批次查看详情
3. 勾选批次进行批量删除或复制
4. 在详情页查看批次内所有记录

## 安全说明

## 待修改内容与修复方案

以下内容来自静态代码审查，属于后续需要优先处理的维护项。

### 1. 修复 tabBar 配置问题

**问题说明**：
- 项目中存在 `miniprogram/custom-tab-bar/index.js`，但 `app.json` 未启用自定义 tabBar。
- 当前原生 `tabBar.list` 中缺少 `text` 字段，可能导致标签文案不显示或配置校验异常。

**修改方案**：
- 如果继续使用原生 tabBar：为首页、历史、设置三个 tab 增加 `text` 字段。
- 如果使用自定义 tabBar：在 `app.json` 的 `tabBar` 中增加 `"custom": true`，并在各页面 `onShow` 中同步当前选中状态。

**建议优先级**：高

### 2. 阻止空扫描批次被保存

**问题说明**：
- 首页点击扫码时会提前创建当前批次。
- 如果用户取消扫码，当前批次可能为空。
- 此时点击“完成本次扫描”可能把空批次写入历史记录。

**修改方案**：
- 将批次创建时机移动到扫码成功或手动输入确认之后。
- 或在 `finishCurrentBatch()` 中增加判断：只有 `items.length > 0` 时才保存。
- 首页“完成本次扫描”按钮应根据当前批次是否有记录显示，而不是只根据最近历史是否存在显示。

**建议优先级**：高

### 3. 修复高级 API 测试预览字段显示

**问题说明**：
- 设置页高级 API 测试区域对 `testResult.parsedResult` 的 WXML 遍历方式不正确。
- 当前显示字段名时使用 `item.label`，但实际遍历的是对象值，可能导致字段名为空或显示错误。

**修改方案**：
- 将 `parsedResult` 转换为数组结构，例如 `{ label, value }[]` 后再渲染。
- 或在 WXML 中使用 `wx:for-index` 获取对象 key，并显示为字段名。

**建议优先级**：高

### 4. 优化高级 API 解析结果的数据模型

**问题说明**：
- 当前接口解析结果以字段显示名作为 key，例如“书名”“作者”。
- 如果用户修改字段显示名，图书详情页可能无法正确读取 `title`、`author` 等标准字段。

**修改方案**：
- 在字段映射中增加标准字段标识，例如 `targetField: title / author / isbn / barcode`。
- `parseResponse()` 同时输出标准字段数据和展示字段数据。
- 图书模式保存记录时使用标准字段，详情展示时使用展示字段。

**建议优先级**：中高

### 5. 在历史详情页展示高级 API 结果

**问题说明**：
- 记录对象中已经保存了 `customResult`、`rawResponse`、`apiConfigId`、`matchedRuleId` 等信息。
- 但历史详情页当前只展示普通扫码内容或少量图书字段，没有完整展示接口解析结果。

**修改方案**：
- 在历史详情页为每条记录增加“接口结果”展示区域。
- 展示解析结果、命中的规则、使用的 API 配置。
- 根据 `showRawResponse` 设置决定是否展示原始响应。

**建议优先级**：中高

### 6. 完善多 API 配置保存与校验

**问题说明**：
- 高级 API 配置支持多个接口，但保存时主要校验当前激活接口。
- 删除接口后，已有规则可能仍绑定到被删除的接口。

**修改方案**：
- 保存高级配置时遍历校验所有 API 配置。
- 删除 API 配置前检查是否有规则绑定该接口。
- 如果存在绑定，应提示用户重新绑定或自动解绑。
- 保存规则时校验 `apiConfigId` 是否真实存在。

**建议优先级**：中高

### 7. 明确多行输入与回车提交的交互关系

**问题说明**：
- 开启“允许换行”后，输入框切换为 `textarea`。
- `textarea` 模式下“回车直接提交”不再生效，容易造成设置项与实际体验不一致。

**修改方案**：
- 开启多行输入时隐藏或禁用“回车直接提交”设置。
- 或在设置页增加说明：多行输入模式下需点击确认按钮提交。

**建议优先级**：中

### 8. 替换项目配置中的占位 appid

**问题说明**：
- `miniprogram/project.config.json` 中的 `appid` 仍为占位值。
- 该配置不能直接用于真实预览、上传或发布。

**修改方案**：
- 本地开发时可继续使用占位配置。
- 准备发布时应替换为真实小程序 appid。
- 如需保护 appid，可使用本地私有配置文件并避免提交敏感配置。

**建议优先级**：中

### 建议修复顺序

1. 修复 tabBar 配置。
2. 阻止空批次保存。
3. 修复高级 API 测试预览字段显示。
4. 完善高级 API 解析结果模型。
5. 在历史详情页展示高级 API 结果。
6. 完善多 API 配置保存、删除和规则绑定校验。
7. 优化多行输入与回车提交的设置联动。
8. 发布前替换真实 appid。
